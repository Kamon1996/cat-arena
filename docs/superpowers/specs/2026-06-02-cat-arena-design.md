# cat-arena — Дизайн-документ

> Дата: 2026-06-02 · Статус: на ревью · Категория контента: **котики** 🐱

## 1. Обзор и цели
Веб-приложение для **попарного голосования** за изображения котиков и их **рейтинга по Glicko-2**.
Пользователи загружают фото котов; посетители в дуэлях 1-на-1 выбирают, кто круче; рейтинг обновляется.
Умный алгоритм подбора пар максимизирует число разных котиков, получающих оценки.

**Ключевые требования:**
- Продуманная, устойчивая к накруткам система рейтинга.
- Алгоритм подбора, дающий оценивать как можно больше разных картинок.
- **Рейтингуется котик** (имя + 1–3 картинки), не отдельная картинка; в дуэли его картинки листаются каруселью.
  Лимиты — **конфигурируемые дефолты**: ≤2 котика на пользователя, ≤3 картинки на котика (меняются позже).
- **Организации (scoped-рейтинг):** котик по коду-приглашению вступает в организацию (≤2 орг на котика); у него
  глобальный рейтинг + локальный рейтинг в каждой орг. Юзер может создать ≤1 организации. Есть отдельный фид батлов
  и лидерборд внутри орг.
- Простой вход; **логин нужен только для загрузки**, голосование анонимно (с заделом закрыть флагом).
- **Сильное SEO.**
- Бесплатный хостинг, но архитектура переживает вирусный всплеск (egress картинок вынесен на R2/CDN).

**Вне MVP (YAGNI):** комментарии, профили с лентой, несколько категорий, мультиязычность, платежи.

**Модули (каждый — одна зона ответственности):**
- `rating/` — чистый Glicko-2 (без БД, юнит-тесты).
- `pairing/` — выбор пары.
- `storage/` — presigned-загрузка в R2, генерация размеров (`sharp`), URL.
- `moderation/` — авто-скрин картинок (Workers AI / NSFWJS) + очередь PENDING/жалоб + действия админа.
- `auth/` — Auth.js (magic-link).
- `org/` — организации: создание (≤1/юзер), код-вступление, org-фид, org-лидерборд, локальный рейтинг.
- `api/` — тонкие хендлеры: получить пару, отправить голос, создать котика с картинками, создать орг, вступить по коду, пожаловаться.
- страницы: `/` (голосование), `/top` (лидерборд), `/cat/[slug]` (SEO-страница), `/upload`, `/org/[slug]` (фид+лидерборд орг), `/orgs/new`, `/admin` (moderator/admin).

## 4. Модель данных (Prisma / Postgres)
```prisma
model Cat {
  id         String   @id @default(cuid())
  slug       String   @unique               // SEO-URL: <name>-<nanoid>
  name       String                         // имя котика (отображение + SEO)
  ownerId    String
  owner      User     @relation(fields: [ownerId], references: [id])
  status     CatStatus @default(PENDING)     // PENDING|ACTIVE|HIDDEN|BANNED
  // Glicko-2 (рейтингуется КОТИК, а не отдельная картинка)
  rating     Float    @default(1500)
  rd         Float    @default(350)
  vol        Float    @default(0.06)
  score      Float    @default(800)          // = rating - 2*rd (денормализовано для сортировки)
  wins       Int      @default(0)
  losses     Int      @default(0)
  draws      Int      @default(0)            // зарезервировано
  timesShown Int      @default(0)
  images     CatImage[]
  reports    Report[]
  orgs       CatOrg[]
  createdAt  DateTime @default(now())
  approvedAt DateTime?
  @@index([status, score])                   // лидерборд
  @@index([status, rd])                       // пул «нуждающихся» для подбора
  @@index([ownerId])                          // лимит ≤2 котика на юзера
}

model CatImage {
  id        String   @id @default(cuid())
  catId     String
  cat       Cat      @relation(fields: [catId], references: [id], onDelete: Cascade)
  r2Key     String                           // оригинал; thumb/card — по соглашению ключей
  width     Int
  height    Int
  position  Int      @default(0)             // порядок в карусели
  status    ImageStatus @default(PENDING)    // модерация/NSFW на уровне картинки
  createdAt DateTime @default(now())
  @@index([catId, position])
}

model Vote {
  id          String   @id @default(cuid())
  winnerCatId String                          // голос — МЕЖДУ КОТИКАМИ
  loserCatId  String
  voterKey    String                          // anonId (cookie) или "user:<id>"
  createdAt   DateTime @default(now())
  @@index([voterKey, createdAt])              // дедуп виденного / история
  @@index([winnerCatId]); @@index([loserCatId])
}

model Report {                                // жалобы на котика
  id        String   @id @default(cuid())
  catId     String
  cat       Cat      @relation(fields: [catId], references: [id], onDelete: Cascade)
  reason    String?
  reporter  String                            // anonId / user
  createdAt DateTime @default(now())
  @@index([catId])
}

enum CatStatus   { PENDING ACTIVE HIDDEN BANNED }
enum ImageStatus { PENDING APPROVED REJECTED }

// User / Account / Session / VerificationToken — стандартные таблицы Auth.js (Prisma adapter).
// У User добавляется поле role:
enum Role { USER MODERATOR ADMIN }            // User.role @default(USER)
```
**Сущность рейтинга — `Cat`** (имя + 1–3 картинки). Голос — между двумя котиками; подделка исключается **подписанным токеном пары** (см. §7).
**Лимиты — конфигурируемые дефолты (константы в конфиге, проверяются в API):** `MAX_CATS_PER_USER = 2`, `MAX_IMAGES_PER_CAT = 3` (меняются позже).
Котик участвует в дуэлях, если `status = ACTIVE` и есть ≥1 картинка `APPROVED`. В дуэли его APPROVED-картинки идут **каруселью** (одна — без карусели; `position` задаёт порядок).

**Организации (scoped-рейтинг):**
```prisma
model Organization {
  id          String   @id @default(cuid())
  name        String   @unique         // обязательное, уникальное
  slug        String   @unique
  description String?                   // опц. описание
  logoR2Key   String?                   // опц. логотип (в R2)
  joinCode    String   @unique         // длинный сгенерированный код-приглашение
  createdById String   @unique         // ≤1 организации на пользователя (уникальность создателя)
  createdAt   DateTime @default(now())
  members     CatOrg[]
}

model CatOrg {                           // членство + ЛОКАЛЬНЫЙ Glicko-2 котика в орг
  catId   String
  orgId   String
  cat     Cat          @relation(fields: [catId], references: [id], onDelete: Cascade)
  org     Organization @relation(fields: [orgId], references: [id], onDelete: Cascade)
  rating  Float @default(1500)
  rd      Float @default(350)
  vol     Float @default(0.06)
  score   Float @default(800)           // лидерборд орг сортируется по score
  wins    Int @default(0)
  losses  Int @default(0)
  draws   Int @default(0)
  timesShown Int @default(0)
  joinedAt DateTime @default(now())
  @@id([catId, orgId])
  @@index([orgId, score])               // лидерборд организации
}
```
**Создание орг:** имя (обязательное, уникальное), опц. логотип (R2) и описание; `joinCode` генерируется автоматически.
**Лимиты:** `MAX_ORGS_PER_USER = 1`, `MAX_ORGS_PER_CAT = 2` (конфигурируемые). **Вступление:** владелец котика
вставляет `joinCode` (при создании или позже) → проверка существования → авто-добавление `CatOrg` (если у котика < 2 орг;
сам валидный код — пропуск, отдельного одобрения владельцем орг не требуется). Котик всегда остаётся и в общем пуле.
Агрегатного «рейтинга самой организации» в MVP нет — только локальный рейтинг котиков (орг-vs-орг см. §16, пост-MVP).

## 5. Рейтинг — Glicko-2
- Три поля на котика: `rating` (μ, старт 1500), `rd` (старт 350), `vol` (σ, старт 0.06).
- После каждой дуэли пересчитываем **обоих** котиков стандартными формулами Glicko-2 (каждый голос = мини-период;
  пол `rd` снизу, чтобы рейтинг оставался отзывчивым).
- Реализация — **чистый модуль `rating/glicko2.ts`** без БД; юнит-тесты на эталонных значениях из работы Гликмана.
- **Лидерборд сортируется по `score = μ − 2·rd`** (нижняя граница 95% ДИ): мало голосов ⇒ большой `rd` ⇒ низкий `score`
  ⇒ нельзя попасть в топ на удаче. `score` денормализован в `Cat` и пересчитывается при каждом голосе.
- **Локальный рейтинг в орг:** при голосе глобальный Glicko-2 обновляется всегда; дополнительно для каждой орг из
  `orgs(W) ∩ orgs(L)` (макс 2) обновляется их `CatOrg`-рейтинг тем же алгоритмом. Org-лидерборд — по `CatOrg.score`.
- **Кнопка «пропустить»** → запрос новой пары, рейтинг и счётчики не меняются (только, опц., лёгкий сигнал подбору).
- Исходы MVP: победа/поражение. Ничьи/«оба плохи» не вводим (поле `draws` зарезервировано).

## 6. Подбор пар
Цели: покрытие (каждый котик набирает оценки), информативность дуэлей, отсутствие повторов для зрителя, дешевизна на масштабе.
1. **Котик A** — взвешенная выборка с приоритетом «не хватает данных»: вес ∝ `rd` и ↓ для часто показанных
   (`1/√(timesShown+1)`). Берём из небольшого пула по индексу `(status, rd)` среди ACTIVE-котиков с ≥1 APPROVED-картинкой — не сортируем всю таблицу.
2. **Котик B** — близкий к A по `score` (`score BETWEEN A.score±окно`, индекс `(status, score)`; `score` монотонен по силе),
   исключая виденных зрителем, `LIMIT` + случайность.
3. **Невиденное:** аноним — `anonId` в подписанной cookie + кольцевой буфер последних ~50 показанных id (без БД-запроса);
   залогиненный — полный дедуп через `Vote`.
4. **ε-случайность** для покрытия и анти-гейминга. Горячие пулы кешируем.
5. **Org-фид (`/org/[slug]`):** та же логика подбора, но пул ограничен членами орг (ACTIVE, ≥1 APPROVED). Голос
   обновляет глобальный рейтинг + локальный рейтинг этой орг (оба котика — её члены по построению).

## 7. Целостность голосов / анти-абьюз
- **HMAC-подписанный токен пары:** при выдаче пары вкладываем подпись `{A, B, nonce, exp}`. Голос принимается только с
  валидным неистёкшим токеном ⇒ нельзя голосовать за самовыбранные пары и слать выдуманные результаты.
- **Одноразовость** токена (nonce + короткий exp).
- **Rate-limit** через **Upstash Redis** (token-bucket по `voterKey`/IP).
- Glicko-2 сам гасит накрутки (устоявшихся почти не сдвинуть).
- **Задел:** эндпоинт голосования принимает и `anonId`, и `userId` ⇒ позже флагом включаем «только залогиненные».
- В токен пары кладём **scope** (`global` или `org:<id>`) — провенанс фида (в org-фиде оба котика по построению члены орг).

## 8. Загрузка: котик + картинки
- Только для залогиненных. Лимиты (конфигурируемые дефолты): `MAX_CATS_PER_USER = 2`, `MAX_IMAGES_PER_CAT = 3`.
- Поток создания котика:
  1. Пользователь задаёт **имя** котика и выбирает 1–3 картинки.
  2. На каждую: `POST /api/upload/sign` → **presigned PUT URL**; клиент грузит **оригинал напрямую в R2** (минуя Vercel).
     Опц. клиентское превью/ресайз (`web-files-image-handling`).
  3. `POST /api/cats` — по каждой картинке сервер берёт оригинал из R2, через **`sharp`** делает `thumb` (~200px) и
     `card` (~800px), **срезает EXIF**, конвертит в **WebP**, пишет в R2; (опц.) NSFW-скрин (§9); создаёт `Cat` + `CatImage[]` (с `position`).
- Тип jpeg/png/webp, ≤10 МБ, макс. сторона. `slug = <name>-<nanoid>`.
- **Код организации (опц.):** поле для `joinCode` при создании или позже (правка котика) → проверка → вступление в орг (≤2).
- Чистка осиротевших: если котик не дозагружен — TTL на временные объекты R2 / отложенная уборка PENDING.

## 9. Модерация — гибрид (авто-скрин + ручная админка)
- **Авто-скрин на загрузке (бесплатно):** каждую `CatImage` прогоняем через **Cloudflare Workers AI**
  (классификатор NSFW + грубая проверка «это кот?»); **NSFWJS** — бесплатный фолбэк, если упрёмся в лимиты Workers AI.
  - явный NSFW ⇒ `CatImage.status = REJECTED`;
  - пограничное / низкая уверенность ⇒ `PENDING` (ручной разбор);
  - чисто ⇒ `APPROVED`.
- **Видимость котика:** `Cat.status` стартует `PENDING`; автоматически → `ACTIVE`, как только у котика есть ≥1
  `APPROVED`-картинка (тогда он сразу попадает в батлы). Если все картинки `PENDING`/`REJECTED` — ждёт админа.
- **Жалобы:** кнопка «пожаловаться» на дуэли и странице котика → строка `Report`; при пороге жалоб котик
  авто-скрывается (`Cat.status = HIDDEN`) до разбора.
- **`/admin`** (роли **MODERATOR** и **ADMIN**): очереди PENDING-картинок/котиков и жалоб; действия approve / reject / hide / delete / ban. **ADMIN** дополнительно: назначение ролей и бан пользователей.
- Пороги авто-скрина и жалоб — конфигурируемые (см. §16).

## 10. Аутентификация
- **Auth.js v5 + Prisma adapter**, сессии в БД.
- **Magic-link** (passwordless): ввод почты → письмо со ссылкой (**Resend** + **React Email**) → вход.
  Подтверждение почты встроено в сам механизм. Опц. позже — вход через Google.
- **Роли:** `USER` (по умолчанию), `MODERATOR` (доступ к модерации), `ADMIN` (всё + назначение ролей и бан). Поле `User.role`.
- **Гейт:** загрузка/создание орг требуют сессии (любой `USER`); `/admin` — только `MODERATOR`/`ADMIN`; голосование анонимно.

## 11. SEO (ключевая цель)
- **Страница котика `/cat/[slug]`** (SSG/ISR): **имя**, карусель его картинок (с `alt`), место в топе, рейтинг, W/L,
  история дуэлей → тысячи long-tail страниц. JSON-LD `ImageObject`/`ImageGallery` + `BreadcrumbList`.
- **Лидерборд `/top`** (ISR): ранжированный список, JSON-LD `ItemList`.
- **Орг-страницы `/org/[slug]`** (фид + лидерборд): индексируем только если в орг ≥ N котиков, иначе `noindex` (чтобы не плодить тонкие страницы).
- **Главная `/`**: интерактивное голосование + SSR-блок ссылок на топ ⇒ краулер находит страницы.
- **`sitemap.xml`** (динамический, все APPROVED) + **`robots.txt`** + canonical.
- **Open Graph / Twitter-карточки** с картинкой котика (виральная петля); динамические OG через `ImageResponse`.
- **Core Web Vitals:** `next/image` (или прямые CDN-URL с предген. размерами во избежание лимита оптимизации Vercel),
  WebP/AVIF, ленивая загрузка, SSG/ISR ⇒ быстрый TTFB.
- Локаль `ru` (задел под hreflang при мультиязыке).

## 13. Обработка ошибок и тесты
- `rating/` и `pairing/` — чистые модули + **Vitest** (эталонные значения Glicko-2, логика покрытия/исключений).
- Вход API/Server Actions/форм — **Zod** на границе; типизированные ошибки; голос идемпотентен (одноразовый токен).
- Загрузка: проверки типа/размера/размерности, обработка сбоев R2, уборка осиротевших.
- Компоненты — **React Testing Library**; главный сценарий (пара → голос → обновление) — **Playwright**.
- Мониторинг: **Sentry** (ошибки/perf), **PostHog** (воронка голосования, популярные котики).
